# AI 海龟汤游戏 PRD（MVP 优化版）

## 1. 产品概述
AI 海龟汤是一个 AI 驱动的推理游戏网站。玩家阅读“汤面”后，通过多轮提问推理故事真相（汤底）。AI 主持人仅回答三种结果：`是`、`否`、`无关`。  
当玩家提交最终猜测或主动结束后，系统揭晓汤底并给出本局结果。

## 2. 产品目标

### 2.1 业务目标
- 在 3 分钟内让新用户完成一局海龟汤体验。
- 提供稳定、低门槛的 AI 互动推理玩法。
- 为后续用户系统、排行榜和多人模式预留扩展空间。

### 2.2 体验目标
- 主持人回答风格统一且可预期（严格三选一）。
- 对话流畅、页面反馈清晰、揭晓汤底有“仪式感”。
- 玩家可完整回顾推理过程。

## 3. 目标用户
- 喜欢推理游戏的玩家。
- 想轻松娱乐的上班族/学生。
- 对 AI 应用感兴趣的技术爱好者。

## 4. 版本范围

### 4.1 MVP（本期必须做）
1. **游戏大厅**
   - 展示可游玩题目列表（支持难度/类型标签）。
   - 支持“开始游戏”进入对应会话。

2. **游戏页面**
   - 展示汤面故事。
   - 提问输入框 + 发送按钮。
   - AI 主持人只返回：`是` / `否` / `无关`。
   - 展示对话历史（问题 + 回答）。
   - 支持”提交最终猜测”。
   - 支持”结束游戏”（等价于主动解锁汤底）。

3. **汤底揭晓页面（或揭晓区块）**
   - 显示完整汤底内容。
   - 显示本局结果：`正确/错误`（当提交最终猜测时）。
   - 显示推理过程（对话历史回放）。
   - 提供“再来一局”入口。

### 4.2 后续迭代（非本期）
- 用户登录系统。
- 跨设备游戏记录保存。
- 排行榜与成就系统。
- 自定义海龟汤（玩家投稿 / AI 出题）。
- 多人对战模式。

## 5. 核心规则定义

### 5.1 术语
- **汤面（Story）**：给玩家看的故事描述，不直接暴露真相。
- **汤底（Truth）**：事件完整真相，用于主持人判定。

### 5.2 AI 回答规则（强约束）
- 仅允许输出：`是`、`否`、`无关`。
- 判定口径：
  - `是`：问题在汤底语义下可判定为真。
  - `否`：问题在汤底语义下可判定为假。
  - `无关`：无法判定或与题目关键事实无关。

### 5.3 解锁规则
- 满足以下任一条件时，解锁汤底：
  1) 玩家提交最终猜测；  
  2) 玩家点击”结束游戏”。

## 6. 用户流程（主链路）
1. 进入大厅，选择题目并开始。
2. 阅读汤面，连续提问。
3. AI 返回三选一答案，玩家继续推理。
4. 玩家提交最终猜测，系统判定正确/错误。
5. 系统展示汤底 + 推理过程 + 再来一局入口。

## 7. 功能需求清单（可验收）

### FR-01 游戏大厅
- 页面展示题目卡片：标题、难度、类型、简介。
- 点击卡片后创建会话并进入游戏页面。
- 验收：点击后 2 秒内有“创建中”反馈，成功后跳转会话页。

### FR-02 游戏会话
- 展示汤面文本。
- 输入问题后可发送，发送后追加到对话区。
- AI 返回仅可为三种答案之一。
- 验收：连续发送 5 次，历史按时间顺序显示，无重复/错位。


### FR-04 最终猜测
- 玩家输入“我认为真相是……”，提交后评估正确/错误。
- 提交成功后解锁汤底并显示结果。
- 验收：提交后页面显示“正确/错误 + 汤底”，且不能重复提交。

### FR-05 结束游戏
- 玩家可主动结束并查看汤底。
- 主动结束后不再允许提问。
- 验收：点击结束后 1 秒内展示汤底并禁用提问区。

### FR-06 推理过程回放
- 汤底揭晓后可回看完整对话历史。
- 验收：历史至少包含“问题 + 回答 + 时间”。

## 8. 界面与视觉要求
- **整体风格**：神秘悬疑，深蓝主色，低饱和高对比。
- **大厅**：卡片式布局，突出难度和题目标签。
- **游戏页**：聊天式布局（类即时通讯），输入区固定底部。
- **揭晓页/区块**：强调“真相揭晓”的仪式感（标题、动效、视觉层次）。

## 9. 技术方案建议

### 9.1 推荐实现（与当前工程一致）
- 前后端同栈：`Next.js + React + TypeScript + Tailwind CSS`
- API：`/api/...` 路由（无需单独 Express 服务）
- AI：优先支持可配置 Provider（Hugging Face / DeepSeek / Claude）
- 部署：Vercel

### 9.2 可选方案（你原模板）
- 前端：React + TypeScript + Tailwind CSS
- 后端：Node.js + Express
- AI：DeepSeek / Claude API
- 部署：Vercel

> 说明：若采用 Express，需要额外处理跨域、部署拓扑和前后端联调复杂度；MVP 推荐同栈以降低交付风险。

## 10. 数据与接口（MVP）

### 10.1 核心数据结构
- `Session`
  - `id`
  - `story`
  - `truth`（仅解锁后返回给前端）
  - `conversation[]`（question/answer/time）
  - `finalGuess`（guess/correct/time）
  - `stage`（questioning/revealed）

### 10.2 API（示例）
- `POST /api/session`：创建会话
- `GET /api/session/:id`：获取会话状态
- `POST /api/session/:id/question`：提交提问，返回三选一
- `POST /api/session/:id/final`：提交最终猜测，返回正确/错误+汤底

## 11. 非功能需求
- **性能**：首屏可交互 < 3s（正常网络），单次问答响应目标 < 5s。
- **可用性**：接口失败时给出明确提示并允许重试。
- **安全性**：汤底在解锁前不可通过前端接口直接读取。
- **可观测性**：记录会话创建失败率、问答失败率、平均提问轮数。

## 12. 风险与对策
- **AI 输出不稳定**：强制结构化输出（JSON）+ 服务端校验与兜底。
- **语义误判**：通过提示词约束与判定策略迭代优化。
- **成本波动**：提供多模型切换与限流策略。
- **并发与状态一致性**：会话状态以服务端为准，关键写操作串行处理。

## 13. 验收标准（MVP 上线门槛）
- 玩家能完整走通：创建会话 → 多轮提问 → 提交最终猜测/达上限 → 查看汤底。
- AI 回答在可见层严格仅出现三种值：`是/否/无关`。
- 解锁规则准确：未解锁前不能查看汤底。
- 对话历史可回看，且顺序正确。
- 主要页面在桌面端与移动端可用。

## 14. 里程碑建议
- **M1**：核心玩法打通（单会话 + 三选一 + 解锁）。
- **M2**：大厅与题库能力（难度/类型）。
- **M3**：体验打磨（视觉风格、动效、错误处理、埋点）。
- **M4**：账号与排行榜（下一期）。
